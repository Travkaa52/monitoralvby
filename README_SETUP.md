# Monitor Kharkiv — обновлённый проект

## 🎨 Новый интерфейс (5 вкладок)
`index.html` и `app.js` переписаны с нуля под макет: тёмный тактический стиль,
радар-виджет, счётчики по типам угроз (Шахеди/Ракети/Авіація), нижняя навигация
с 5 вкладками:
- **Карта** — сама карта + статистика + баннер последней угрозы + статус ППО.
- **Тривоги** — лента активных/исторических тревог, тап открывает детальную карточку
  (тип, направление, вероятная цель, источник, уровень опасности).
- **Канали** — список каналов из `channels.json` с переключателями "показывать/скрыть"
  (работает только локально на устройстве — см. пояснение ниже) и вкладка
  "Додати канал" для личной пометки.
- **Історія** — обезвреженные/просроченные цели (хранится в localStorage браузера).
- **Налаштування** — звук/вибрация, фильтры по типам целей, интервал фонового
  обновления, очистка истории.

**Важно про `channels.json` и вкладку "Канали":** это чисто фронтенд-список для
отображения и локальной фильтрации на конкретном устройстве. Он не читает и не
меняет секрет `SOURCE_CHANNELS`, который реально определяет, какие Telegram-каналы
парсит `parser.py` — эти списки нужно поддерживать в соответствии вручную. Отредактируй
`channels.json`, вписав туда реальные каналы из `SOURCE_CHANNELS`, чтобы вкладка
показывала актуальный список.

## ⚡ Важно про "реальное время" на GitHub Actions
GitHub Actions **не запускает cron ровно по расписанию** — при высокой нагрузке на
инфраструктуру GitHub задержка запуска `*/5 * * * *` может составлять от секунд до
10-15 минут, это ограничение платформы, а не этого кода. Кроме того, GitHub
автоматически **отключает scheduled workflow**, если в репозитории 60 дней не было
пушей — придётся руками зайти в Actions → Enable workflow. Если нужна честная
задержка в единицы секунд — это уже не GitHub Actions, а постоянно работающий
процесс на отдельном сервере/VPS/Railway (как сейчас работает `bot.py`, который в
принципе не может жить внутри Actions, т.к. Actions — это "включить/сделать
работу/выключить", а не долгоживущий процесс).

Что реально было переделано, чтобы уложиться в эту модель максимально надёжно:

## Что изменилось (эта правка)
- **parser.py — переписан на "догоняющий" опрос вместо живого прослушивания.**
  Раньше парсер слушал Telegram события 280 секунд внутри 5-минутного окна и
  выключался — все сообщения, пришедшие в оставшиеся ~20 секунд (а тем более если
  GitHub Actions опоздал с запуском на пару минут), терялись безвозвратно. Теперь
  при каждом запуске парсер запоминает `last_message_id` по каждому каналу в
  `parser_state.json` и на следующем запуске подтягивает **всю** историю с этого
  id — сообщения больше не теряются, даже если Actions задержался или совсем
  пропустил цикл. Сам проход теперь занимает секунды, а не 280 секунд.
- **targets.json был пустым файлом (0 байт)** — это невалидный JSON, из-за чего
  `fetch('targets.json')` на фронтенде падал с ошибкой синхронизации. Исправлено
  на `[]`.
- **types.json был невалидным JSON** (пропущенные запятые, пустая строка как
  ключевое слово) — keyword-фолбек классификации фактически не работал вообще,
  если Gemini недоступен. Исправлено.
- Убрано ключевое слово `"ціль"` из типа `missile` — оно слишком общее и
  превращало почти любое сообщение с этим словом в "ракету".
- **admin_panel.yml писал в формате `{"items":[...]}`**, а парсер и фронтенд
  используют плоский массив `[...]` — ручные метки через админ-панель фактически
  ломали файл для фронтенда. Приведено к единому формату.
- Добавлен `concurrency`-guard в `main.yml`, чтобы два параллельных запуска
  парсера не толкались за git push.
- Добавлен `MAX_MESSAGE_AGE_MIN` — старые сообщения (например, при первом запуске
  для канала) больше не всплывают на карте как "новые" цели.

## Что изменилось (предыдущая правка)
- **parser.py** — теперь слушает несколько каналов через `SOURCE_CHANNELS` (env), никаких токенов в коде.
- **ai_classifier.py** (новый) — отправляет текст сообщения в Gemini и получает структурированный JSON: тип цели, населённый пункт, направление движения и азимут (0-360°). Если `GEMINI_API_KEY` не задан или запрос упал — парсер автоматически откатывается на старый keyword-поиск, так что система не падает без ИИ.
- **app.js** — маркеры на карте теперь поворачиваются по азимуту цели (плагин `leaflet-rotatedMarker`, который уже был подключён), плюс направление показывается в попапе маркера.
- **bot.py** (новый) — бот на aiogram: Menu Button + inline-кнопка открывают карту как Telegram Mini App.
- **index.html** — исправлен сломанный дублирующийся скрипт (там был реальный синтаксис-баг), добавлен Telegram WebApp SDK.
- **requirements.txt** — добавлен aiogram.
- **.gitignore** — теперь реально исключает `.env` и файлы сессий Telethon.
- Из архива удалены `parser.py.bak`, `deepseek_html_*.bak` и `bot_session.session*` — в них были твои реальные `API_HASH`, `BOT_TOKEN` и GitHub PAT в открытом виде, и живая Telegram-сессия. **Их нужно перевыпустить/отозвать, если этот проект хоть раз попадал в публичный репозиторий.**

## Как работает AI-классификация
1. Приходит новое сообщение из любого канала в `SOURCE_CHANNELS`.
2. `parser.py` отправляет текст в Gemini (`ai_classifier.analyze_message`) с чёткой схемой ответа (JSON: `type`, `location`, `direction_text`, `bearing_degrees`, `confidence`).
3. Каждая найденная цель с `confidence >= 0.4` сопоставляется с известным населённым пунктом (точное совпадение → нечёткое совпадение через `difflib` → подстрока — на случай опечаток или иной формы слова).
4. Азимут (`bearing_degrees`) идёт в `targets.json` как `bearing` — фронтенд поворачивает иконку в эту сторону. Если Gemini не дал число, но есть текстовое направление ("на схід", "у бік Харкова") — азимут прикидывается по словарю сторон света (грубый фолбек).
5. Если Gemini недоступен (нет ключа, лимит, сетевая ошибка) — используется старый keyword-поиск по спискам `TARGET_TYPES`, без направления. Админам (`ADMIN_IDS`) уходит уведомление о переходе на фолбек.

Получить бесплатный `GEMINI_API_KEY` можно в Google AI Studio (aistudio.google.com) — просто сгенерировать ключ, без привязки карты для базового бесплатного лимита.

## ⚠️ Обязательно сделать прямо сейчас
1. Перевыпусти `API_HASH` на my.telegram.org (или как минимум проверь, не утекал ли он).
2. Отзови старый `BOT_TOKEN` через @BotFather → `/revoke`, получи новый.
3. Удали/отзови GitHub Personal Access Token, который был в parser.py.bak (Settings → Developer settings → PAT).

## GitHub Secrets (Settings → Secrets and variables → Actions)
- `API_ID`, `API_HASH` — с my.telegram.org
- `SOURCE_CHANNELS` — например `monitorkh1654,channel2,channel3`
- `BOT_TOKEN` — токен бота (используется и парсером, и для Mini App); бот должен
  быть админом в каждом канале из `SOURCE_CHANNELS`
- `GEMINI_API_KEY` — (опционально, но нужен для AI-классификации) ключ из Google AI Studio
- `ADMIN_IDS` — (опционально) твой user_id для алертов о падении парсера/AI

`GITHUB_TOKEN` вручную создавать не нужно — Actions подставляет его сам, и в
`main.yml` уже включено `permissions: contents: write`, этого достаточно для
коммита `targets.json`.

## Локальный запуск бота (bot.py)
GitHub Actions не подходит для постоянно работающего бота (это cron, а не долгоживущий процесс).
Бота нужно держать где-то отдельно — на своём сервере / Railway / PythonAnywhere / VPS:

```bash
pip install -r requirements.txt
export BOT_TOKEN="..."
export WEBAPP_URL="https://<user>.github.io/<repo>/"
python bot.py
```

После этого у пользователей в чате с ботом появится:
- кнопка меню "Карта загроз" рядом с полем ввода,
- и кнопка "🗺 Открыть карту загроз" под `/start`.

Обе открывают `index.html` как Telegram Mini App (внутри Telegram, с автоматической темой оформления).

## 🕒 Постійний (24/7) режим — свій акаунт, без адмінки в каналі

GitHub Actions для цього не підходить (це cron з затримками й лімітом 60 днів
без пушів — див. вище). Для чесного 24/7 без затримок потрібен окремий
довгоживучий процес на сервері/VPS (Railway, PythonAnywhere, звичайний Linux-сервер
і т.д.), який просто ніколи не вимикається.

Головна відмінність від старого "бот"-режиму: тут використовується твій **звичайний
Telegram-акаунт** через `API_ID`/`API_HASH` (Telethon user-сесія), а не `BOT_TOKEN`.
Це означає:
- **не потрібно додавати нікого адміністратором** у канал — досить, щоб канал був
  публічним (або щоб сам акаунт був у ньому підписаний, якщо приватний);
- акаунт має повний доступ до `GetHistory`, тому при старті парсер сам підтягує все,
  що прийшло за час простою (доганяючий прохід), а потім слухає нові повідомлення
  без обмеження часу — нічого не губиться навіть при перезапуску сервера.

### Крок 1 — отримати SESSION_STRING (один раз)
```bash
pip install -r requirements.txt
python generate_session.py
```
Скрипт спитає `API_ID`/`API_HASH` (з my.telegram.org), номер телефону і код із
Telegram (і пароль 2FA, якщо є) — і один раз надрукує `SESSION_STRING`. Збережи
його як секрет — далі повторно логінитись не треба.

⚠️ `SESSION_STRING` — це фактично доступ до акаунта без пароля. Ніколи не комить
його в git і не показуй нікому.

### Крок 2 — запустити постійний процес
```bash
export API_ID="..."
export API_HASH="..."
export SESSION_STRING="..."       # отриманий на кроці 1
export SOURCE_CHANNELS="monitorkh1654,channel2,channel3"
export FOREVER=1                  # вмикає 24/7-режим замість одноразового
export ADMIN_IDS="твій_user_id"   # опційно — сповіщення при падінні (потрібен ще BOT_TOKEN)
export BOT_TOKEN="..."            # опційно, лише для notify_admins, не для читання каналів
python parser.py
```
Процес підключається, робить доганяючий прохід і далі слухає безкінечно. При обриві
мережі чи помилці — сам чекає `RECONNECT_DELAY_SEC` (за замовч. 15с) і перепідключається,
не завершуючи процес.

Якщо `SESSION_STRING` не задавати — Telethon один раз спитає номер телефону/код
прямо в консолі і збереже сесію локально у файл `user_session.session` (не комить
його в git, він уже в `.gitignore`). Для сервера без інтерактивного доступу
(Docker, CI) `SESSION_STRING` — обов'язковий.

### Крок 3 — щоб процес не падав разом із SSH-сесією
Найнадійніше — systemd-юніт (приклад лежить у `monitoralvby.service`):
```bash
sudo cp monitoralvby.service /etc/systemd/system/
sudo mkdir -p /opt/monitoralvby && cp -r . /opt/monitoralvby
# впиши API_ID/API_HASH/SESSION_STRING/SOURCE_CHANNELS у /opt/monitoralvby/.env
sudo systemctl daemon-reload
sudo systemctl enable --now monitoralvby
sudo systemctl status monitoralvby      # перевірити, що працює
journalctl -u monitoralvby -f           # живі логи
```
`Restart=always` у юніті сам піднімає процес, якщо він все ж впаде або сервер
перезавантажиться. Альтернатива без systemd — `tmux`/`screen`/`nohup`, але вони
не переживають перезавантаження сервера.

Якщо потрібен саме старий одноразовий режим (GitHub Actions) — просто не задавай
`FOREVER=1`, і `parser.py` продовжить працювати як раніше (`run_once()`).

### GitHub Actions замість власного сервера — вікна по 5 годин
Якщо немає окремого сервера/VPS, можна імітувати 24/7 прямо в GitHub Actions —
файл `.github/workflows/monitor_247.yml` вже це робить:
- крон `0 */5 * * *` перезапускає джобу кожні 5 годин (00:00, 05:00, 10:00,
  15:00, 20:00 UTC — зі звичайною для Actions похибкою в кілька хвилин);
- усередині кожного вікна `parser.py` працює в постійному режимі
  (`FOREVER=1`, `FOREVER_MAX_MINUTES=295`) і сам акуратно завершується
  трохи раніше за ліміт джоби (360 хв), а не обривається примусово;
- кожне побачене повідомлення одразу пишеться у `messages_log.jsonl`
  (`log_raw_message`) — це журнал усього, що бачив парсер, а не лише
  розпізнаних цілей;
- `AUTO_GIT_COMMIT=1` + `COMMIT_INTERVAL_SEC=300` — сам процес коммітить і
  пушить `targets.json` / `parser_state.json` / `messages_log.jsonl` кожні
  5 хвилин прямо під час роботи, а не тільки в кінці — якщо GitHub все ж
  вб'є джобу достроково, втратиться щонайбільше 5 хвилин даних.

Обов'язково додай секрет `SESSION_STRING` (отримай через `generate_session.py`
локально один раз) — без нього в CI логін неможливий (нема кому вводити код
з Telegram інтерактивно).

Старий `.github/workflows/main.yml` (крон раз на 5 хвилин, одноразовий
доганяючий прохід) залишений, але його розклад вимкнено — вмикати обидва
одночасно не можна: вони пишуть і коммітять ті самі файли й почнуть штовхатись.
Якщо потрібен саме старий режим — прибери коментар з `cron` у `main.yml` і
вимкни (постав тільки `workflow_dispatch`) `monitor_247.yml`.


Парсер теперь **не пушит ничего в GitHub сам** — он просто один раз проходит по
каналам и обновляет локальные `targets.json` и `parser_state.json`. Коммит и push
делает шаг `Commit and push` в `main.yml`. Для локального запуска GitHub вообще не
нужен:

```bash
export API_ID="..."
export API_HASH="..."
export SOURCE_CHANNELS="monitorkh1654,channel2"
export BOT_TOKEN="..."
python parser.py
```

При первом запуске Telethon создаст `parser_session.session` — этот файл никогда
не коммить (уже в `.gitignore`). Обязательное условие: бот, чей `BOT_TOKEN`
используется, должен быть добавлен **администратором** в каждый канал из
`SOURCE_CHANNELS` — иначе Telegram не даст ни слушать, ни читать историю канала.

## Публикация карты через GitHub Pages
1. Settings → Pages → Source: `Deploy from a branch` → branch `main`, папка `/ (root)`.
2. Через минуту карта будет доступна на `https://<user>.github.io/<repo>/`.
3. Отдельный workflow для этого не нужен — `index.html` лежит в корне репозитория,
   Pages отдаёт статику напрямую, а `app.js` каждые 5 секунд опрашивает
   `targets.json`, который обновляет `main.yml`.
